# Работа с LLM при разработке: краткий гайд для команды

Внутренний материал для передачи опыта. LLM — не замена разработчику, а ускоритель при хорошем процессе: ясные цели, уточняющие вопросы, согласование архитектуры, зафиксированный контекст проекта и обязательная проверка результата.

---

## Как устроен процесс (9 блоков)

| Блок | Суть |
|------|------|
| **1. Диалог** | Говорите понятно, не стесняйтесь переспрашивать |
| **2. Критическое мышление** | Проверяйте решения и «стоимость вопроса» |
| **3. Актуальность** | Версии библиотек и API — перепроверять |
| **4. Архитектура** | Согласовывать до крупных изменений |
| **5. Среда и проект** | Настраивать репозиторий под совместную работу |
| **6. Верификация** | Тесты, дифф, ответственность — на человеке |
| **7. Режимы работы** | Спек для фич, ad-hoc для хотфиксов — с явным порогом перехода |
| **8. Управление сессией** | Когда остановиться, как не утонуть в каскаде фиксов |
| **9. Работа соло** | Компенсация отсутствия ревьюера процессом |

## Чеклист перед задачей

- [ ] Сформулировал цель **живым языком**, но с явными границами (что делать / чего не трогать)
- [ ] Назвал ограничения: объёмы данных, сроки, платформы разработки, язык документации, i18n
- [ ] Уточнил, **понял ли ассистент задачу так же**, как я (пересказ своими словами)
- [ ] Спросил: **есть ли альтернатива** и какова «стоимость вопроса» (сложность, lock-in, сопровождение)
- [ ] Задал **свои доменные вопросы** (модель часто не знает ваших нюансов)
- [ ] Для внешних зависимостей: **актуальные версии и API** (документация, репозиторий, поиск)
- [ ] Обсудил **архитектуру** до большого диффа; разбил задачу на шаги
- [ ] Среда позволяет **запускать проверки** (сборка, тесты, линтер) одной-двумя командами
- [ ] Повторяющийся контекст проекта **закреплён в репозитории** (правила, AGENTS.md, скрипты), а не только в чате
- [ ] Указан **уровень риска**: черновик / прод / security-critical

## Чеклист после ответа модели

- [ ] Прочитал **дифф**, а не только текст в чате
- [ ] Запустил **тесты и линтер** (или ручной сценарий для UI)
- [ ] Проверил: нет ли лишних файлов, секретов, случайных зависимостей
- [ ] Если что-то непонятно — **попросил объяснить**, прежде чем мержить
- [ ] Крупное изменение — **второй взгляд** коллеги или отдельное ревью

## Практики по блокам

### 1. Диалог

- Общайтесь **естественным языком** — модели обучены на человеческой речи.
- «Натуральный» ≠ «расплывчатый»: добавляйте конкретику там, где важны границы.
- Не стесняйтесь: *«Я не понял, объясни, как это работает»*.
- Повторяйте вопросы, пока не сойдётесь в понимании **и** в выборе подхода.

### 2. Критическое мышление

- Требуйте обоснование: *почему эта библиотека, а не другая?*
- Оценивайте «стоимость вопроса»: миграции, производительность, зависимости, сопровождение.
- Задавайте вопросы, которые **важны вам**: объёмы данных, SLA, legacy, политики компании.

### 3. Актуальность

- Модель может уверенно предложить **устаревший или несуществующий API**.
- Явно просите свериться с документацией / кодовой базой / актуальными релизами.
- Всё равно **прогоняйте код** — рассуждение в чате не заменяет запуск.

### 4. Архитектура

- Обсуждайте структуру **до** массовых правок.
- Предпочитайте несколько **маленьких согласованных шагов** одному «сделай всё».
- Фиксируйте решения в короткой заметке или ADR, если выбор долгоживущий.

### 5. Среда и проект

- Настраивайте окружение для **пары «человек + модель»**, не только под себя:
  - единые команды (`task test`, `npm run lint`, `pytest`);
  - правила проекта в файлах репозитория;
  - доступ к терминалу, git, структуре кода.
- Минимум расширений IDE — меньше шума и конфликтов.
- Рабочее место = **диалог + ревью диффа + запуск проверок**, не только окно промпта.
- При настройке инструментов обсудите с моделью: мультиплатформенность **разработки**, i18n, язык и формат документации.
- Периодически **пересматривайте практики**: облачные модели и инструменты меняются чаще, чем кажется.

### 6. Верификация и ответственность

| Человек | Модель |
|---------|--------|
| цель, приоритеты, «зачем» | черновики, варианты, рутина |
| ограничения и риски | объём кода, рефакторинг |
| ревью диффа, финальное «мержим / нет» | поиск по кодовой базе, итерации по ошибкам |
| ответственность за прод | — |

- Правило: **не принято, пока не зелёные проверки** (или явно задокументированное исключение).
- Не вставляйте секреты в чат; осторожно с новыми зависимостями «по совету модели».

### Когда LLM — плохой основной инструмент

- Юридически значимые формулировки без юриста.
- Safety-critical логика без формальной верификации.
- Расследование инцидента без логов и фактов — нужен человек и процесс.

## Два режима работы: спек vs. ad-hoc

На практике работа с LLM делится на два принципиально разных режима. Оба легитимны — но требуют разной дисциплины.

### Режим «Спек» (плановая фича)

Полный цикл: requirements → design → tasks → реализация по шагам → property-тесты → коммит с конвенциональным сообщением.

**Когда использовать:**
- Новый функционал с 3+ файлами изменений
- Изменения в архитектуре или схеме данных
- Фичи, которые затрагивают и бэкенд, и фронтенд
- Всё, что потребует объяснения коллегам через неделю

**Характеристики:**
- Формальные acceptance criteria (SHALL/WHEN/THEN)
- Атомарные задачи с чекпоинтами
- Property-тесты как критерий приёмки
- Feature-ветка от main

### Режим «Ad-hoc» (хотфикс, эксперимент)

Быстрая правка без формального спека: описание проблемы → фикс → проверка → коммит.

**Когда допустим:**
- Баг, локализованный в одном файле/методе
- Исправление после деплоя (hotfix)
- Эксперимент, который можно откатить одним `git revert`
- Косметика: опечатки, порядок импортов, мелкий рефакторинг

**Минимальный чеклист ad-hoc режима:**
- [ ] Описал проблему в одном предложении (что сломано, где)
- [ ] Убедился, что фикс не затрагивает соседние модули (дифф < 50 строк)
- [ ] Запустил тесты / сборку
- [ ] Коммит-сообщение содержит **что** и **почему**, даже если коротко (`fix: normalize path separators in export — backslash from Windows imports`)
- [ ] Если фикс вырос в 3+ файла — остановился и перешёл в режим «Спек»

### Порог перехода между режимами

| Признак | Ad-hoc достаточно | Нужен спек |
|---------|-------------------|------------|
| Файлов затронуто | 1–2 | 3+ |
| Есть миграция БД | Нет | Да |
| Затрагивает API-контракт | Нет | Да |
| Требует новых зависимостей | Нет | Да |
| Нужно будет объяснять | Нет | Да |
| Время реализации | < 30 мин | > 30 мин |

Если при работе в ad-hoc режиме обнаруживаете, что уже третий файл открыт — это сигнал остановиться, создать спек и перейти в плановый режим.

## Управление сессией: когда остановиться

Длинные сессии с LLM создают иллюзию прогресса. После 2–3 часов непрерывной работы внимание к диффам падает, а порог «ну ладно, выглядит нормально» снижается. Серия из 5+ последовательных фиксов за вечер — симптом того, что первый коммит был принят без достаточной проверки.

### Правила управления длиной сессии

**Сигналы «пора остановиться»:**
- Третий `fix:` коммит подряд к одной и той же фиче
- Желание написать коммит-сообщение `fixes` без уточнения
- Обнаружение бага, который вы сами создали 20 минут назад
- Потеря фокуса: не можете объяснить, что именно делает последний дифф
- Сессия длится > 3 часов без перерыва

**Что делать:**
- Зафиксировать текущее состояние (коммит WIP или stash)
- Записать одно предложение: «где остановился, что дальше»
- Вернуться после паузы и начать с ревью собственного диффа за сессию

**Планирование блоков:**
- Один «спек-блок» = 1–2 часа целенаправленной работы
- Между блоками — пауза или переключение контекста
- Ежедневный лимит: 2–3 блока плановой работы с LLM, затем — асинхронные задачи или ревью

### Антипаттерн: «ещё один фикс»

Самый частый сценарий деградации:
```
коммит feat → обнаружен баг → fix → ещё баг → fix → ещё → fix → fix → fix
```

Причина: первый коммит был принят без достаточной проверки (тесты не покрыли edge case, дифф просмотрен бегло). Каждый следующий фикс — это долг за пропущенную верификацию.

Решение: если второй фикс к одному коммиту потребовался в течение часа — остановиться, откатить до стабильного состояния, понять корневую причину, и переделать первый коммит нормально.

## Работа соло: компенсация отсутствия ревьюера

Когда нет коллеги для code review, нужно компенсировать это процессом.

### Стратегии

**1. «Утро после»**
Не мержить feature-ветку в тот же день. Утром следующего дня — ревью собственного PR с «холодной головой». Часто находятся вещи, незаметные в потоке.

**2. Отложенный self-review**
После завершения блока работы — `git diff main..HEAD` целиком. Читать как чужой код: понятны ли имена, нет ли лишнего, согласованы ли изменения между собой.

**3. LLM как ревьюер (с оговорками)**
Можно попросить модель в отдельной сессии проревьюить дифф — но помнить ограничения:
- Модель не знает ваш бизнес-контекст
- Модель может пропустить логические ошибки, которые формально «компилируются»
- Это не замена ревью, а дополнительный сигнал

**4. Property-тесты как «ревьюер поведения»**
Если человеческого ревью нет — автоматические тесты становятся критически важны. Property-тесты (не только unit) покрывают сценарии, о которых вы могли не подумать.

**5. Чеклист перед мержем (соло)**
- [ ] Прошло > 2 часов с момента последнего коммита в ветке (остыл)
- [ ] `git diff main..HEAD` прочитан целиком
- [ ] Тесты зелёные
- [ ] Коммит-сообщения понятны без контекста чата
- [ ] Нет файлов, которые «забыл убрать» (debug-логи, TODO-хаки, временные зависимости)

## Примеры фраз для промптов

**Уточнение понимания**

> Правильно ли я понимаю: ты предлагаешь кэшировать X в Redis, потому что Y, а вариант с in-memory мы откладываем из-за Z?

**Границы задачи**

> Добавь endpoint для экспорта заметок. Не меняй схему БД и не трогай фронтенд. Сначала план в 5 пунктах, потом код.

**Альтернативы и стоимость**

> Сравни библиотеки A и B для нашего случая: 500 ГБ данных, nightly batch, Python 3.13. Что проще сопровождать через 2 года?

**Доменный нюанс**

> Учти, что исходные файлы — на NAS, путь только read-only, а индекс в PostgreSQL. Как это меняет план?

**Актуальность API**

> Проверь сигнатуры по актуальной документации FastAPI 0.115 и нашему `pyproject.toml`. Не используй устаревшие параметры.

**Архитектура до кода**

> Прежде чем писать код: опиши компоненты, потоки данных и точки отказа. Жду согласования — потом реализация по шагам.

**Объяснение**

> Объясни, как работает этот патч, как для коллеги, который не видел контекст. Где основной риск регрессии?

**Верификация**

> После изменений запусти тесты и линтер. Если что-то падает — исправь и кратко отчитайся, что именно сломалось.

**Уровень риска**

> Это черновик для эксперимента: можно упростить, тесты минимальные. / Это меняет авторизацию: нужен консервативный план и полное покрытие тестами.

## Антипаттерны

| Антипаттерн | Почему плохо | Что делать вместо |
|-------------|--------------|-------------------|
| Один огромный промпт «переделай всё» | Регрессии, несогласованная архитектура | Дробить на шаги с чекпоинтами |
| Верить красивому объяснению без запуска | Модель убедительна даже в ошибке | Тесты, сборка, ручной сценарий |
| Смотреть только чат, не дифф | Ломается соседний код | Ревью каждого изменённого файла |
| Канцелярит и пустые абстракции | Модель не понимает приоритеты | Живой язык + явные ограничения |
| Весь контекст только в голове | Модель не угадает терабайт и NAS | Задавать свои вопросы и фиксировать в репо |
| Зоопарк IDE-расширений | Шум, конфликты, другой контекст | Минимум; фокус на проекте и проверках |
| «Модель работает, я отдыхаю» | Ответственность за прод на человеке | Диалог + направление + верификация |
| Слепое копирование зависимостей | Supply-chain и лишний вес | Явно спросить про альтернативы и риски |
| Секреты в промпте | Утечка ключей | `.env`, переменные окружения, политика фирмы |
| Спорить с моделью без фактов | Пустая трата времени | Логи, тест, документация — потом уточнение |
| Коммит «fixes» без контекста | Невозможно понять через неделю; сложно откатить конкретное изменение | Даже в ad-hoc: `fix: что и почему` в одну строку |
| Марафон 20+ коммитов за день | Деградация внимания → каскад фиксов → технический долг | 2–3 блока по 1–2 часа с паузами |
| Ad-hoc на 5+ файлов | Это уже не хотфикс, а незапланированная фича без спека | Остановиться, создать спек, перейти в плановый режим |
| Мержить ветку в день создания (соло) | Нет «холодной» проверки | Отложить PR на «утро после» |

## Минимальный рабочий цикл (одна задача)

### Режим «Спек» (плановая фича)

```
1. Цель + границы + ограничения → requirements.md
2. Дизайн / архитектура → design.md → согласование
3. Декомпозиция → tasks.md (атомарные шаги + чекпоинты)
4. Реализация по одной задаче за раз
5. Дифф + тесты + линтер после каждого шага
6. Чекпоинт: всё зелёное → следующий блок
7. Непонятно → «объясни» → при необходимости откат / правка
8. Коммит с конвенциональным сообщением
```

### Режим «Ad-hoc» (хотфикс)

```
1. Проблема в одном предложении (что, где, как воспроизвести)
2. Фикс → дифф < 50 строк, 1–2 файла
3. Тесты зелёные
4. Коммит: fix: что и почему
5. Если вырос в 3+ файла → СТОП → создать спек
```

## Одна фраза для руководства

**Эффективная работа с LLM — это дисциплина диалога и проверки, а не магия промпта:** чем лучше вы формулируете цель, ловите слепые зоны и прогоняете проверки, тем меньше переделок и тем безопаснее ускорение разработки.
